## What is reQuery?

reQuery is a **jQuery 4 plugin** that adds reactive state and declarative DOM binding to jQuery-based projects.

You initialize state on a DOM element, annotate your HTML with `data-rq-*` attributes, and reQuery keeps the two in sync. When state changes, the DOM updates. When inputs change, state updates. No manual `$('.counter').text(count)` calls scattered through your event handlers.

```js
$('#app').rqState({ count: 0 });

$('#inc').on('click', () => $('#app').rqMutate('count', n => n + 1));
```

```html
<span data-rq-text="count"></span>
```

That's the whole model.

## Why does it exist?

Most reactive UI libraries — React, Vue, Svelte — are designed around a **component tree**. You adopt one as your entire rendering strategy. They're powerful, but they carry significant weight: a build pipeline, a new mental model, and a hard break from any existing jQuery-based code.

A lot of projects don't need all of that. They have:

- An existing jQuery codebase that works
- Small interactive widgets — a counter, a filterable list, a live form
- A design team that owns the HTML and doesn't want it generated by JavaScript
- A preference for dropping a `<script>` tag rather than running a bundler

reQuery is for those projects. It brings the *ideas* of reactivity — state, binding, watchers, derived values — without the framework adoption cost.

## What it is not

reQuery is not a framework and deliberately avoids framework-shaped ideas:

- **No component tree.** There are no parent/child components, no slots, no props passing between components.
- **No virtual DOM.** Updates are direct jQuery/DOM calls. What you see in DevTools is what's there.
- **No lifecycle hooks.** There's no `mounted`, `unmounted`, `beforeUpdate`, or equivalent.
- **No global store.** State is scoped to a DOM element, not shared across a Pinia/Redux-style store.
- **No build step required.** Drop `requery.umd.js` next to jQuery in a `<script>` tag and you're done.

<Aside type="tip">
  If you find yourself wanting components, a router, or a global store — those are signs your project has outgrown reQuery's scope. That's not a failure; it's the right time to reach for Vue or React.
</Aside>

## Who is it for?

- **jQuery developers** who want reactivity without leaving the jQuery model
- **Teams with existing jQuery codebases** that need to incrementally modernise specific UI sections
- **Projects that serve plain HTML** (CMS-driven sites, server-rendered apps) and need a lightweight layer of interactivity
- **Developers who value simplicity** and want to understand the full stack without reading a compiler's output

## How it fits in

reQuery works **alongside** jQuery — it doesn't replace it. Standard jQuery code continues to work exactly as before. You opt specific elements into reQuery's reactive model by calling `rqState()` on them. Everything else on the page is untouched.

```js
// Plain jQuery — still works exactly as before
$('.menu-toggle').on('click', function () {
  $(this).toggleClass('open');
});

// reQuery — reactive state on a specific widget
$('#cart-widget').rqState({ itemCount: 0, total: 0 });
```
