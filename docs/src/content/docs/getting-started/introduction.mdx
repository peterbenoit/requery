## What is reQuery?

reQuery is a **jQuery 4 plugin** that adds reactive state and declarative DOM binding to jQuery-based projects.

You initialize state on a DOM element, annotate your HTML with `data-rq-*` attributes, and reQuery keeps the two in sync. When state changes, the DOM updates. When inputs change, state updates. No manual `$('.counter').text(count)` calls scattered through your event handlers.

```js
$('#app').rqState({ count: 0 });

$('#inc').on('click', () => $('#app').rqMutate('count', n => n + 1));
```

```html
<span data-rq-text="count"></span>
```

That's the whole model.

## Why not just use jQuery?

You can. And for simple cases, you should. But vanilla jQuery is **imperative** — you tell it what to do, not what things should look like. That works fine until state gets involved:

```js
// Every time count changes, you have to remember every place it's displayed
$('#inc').on('click', function () {
  count++;
  $('.counter-display').text(count);       // update the display
  $('.step-indicator').text(`Step ${count}`); // oh, and this one
  if (count > 9) $('.warning').show();     // and this condition
  updateProgressBar(count);               // and this function
});
```

This is fine with two elements. At ten, it's a maintenance problem — every time you add something to the UI that depends on `count`, you find every event handler that changes it and add another line. Miss one and you have a bug.

reQuery inverts this. State is the source of truth; the HTML declares what depends on it:

```html
<span data-rq-text="count"></span>
<span data-rq-text="count" id="step-indicator"></span>
<div data-rq-show="isOverLimit"></div>
```

```js
$('#inc').on('click', () => $('#app').rqMutate('count', n => n + 1));
```

Adding another element that reflects `count` means adding an attribute to the HTML — not finding and editing event handlers.

## What about React, Vue, or Svelte?

Those are excellent tools for the problems they're designed to solve. reQuery doesn't compete with them — it occupies a different position:

| | reQuery | React / Vue / Svelte |
|---|---|---|
| **Adoption** | Add to any page, opt in per element | Replaces your rendering layer |
| **Build step** | None required | Required |
| **Existing jQuery code** | Coexists unchanged | Requires migration |
| **Mental model** | jQuery + HTML attributes | Components, JSX/templates, reactivity graph |
| **Right for** | Widgets, progressive enhancement, CMS pages | SPAs, complex UIs, large teams |

If you're starting a new, complex UI from scratch — use React or Vue. If you have an existing site and want to make parts of it reactive without a rewrite, reQuery is the shorter path.

## Why does it exist?

A lot of web projects live in a middle ground: too much interactivity for plain jQuery event handlers, not enough to justify adopting a full component framework. reQuery is built for that space — it brings reactivity to projects that already speak jQuery, without asking them to change everything else.

## Who is it for?

- **jQuery developers** who want reactivity without leaving the jQuery model
- **Teams with existing jQuery codebases** that need to incrementally modernise specific UI sections
- **Projects that serve plain HTML** (CMS-driven sites, server-rendered apps) and need a lightweight layer of interactivity
- **Developers who value simplicity** and want to understand the full stack without reading a compiler's output

## How it fits in

reQuery works **alongside** jQuery — it doesn't replace it. Standard jQuery code continues to work exactly as before. You opt specific elements into reQuery's reactive model by calling `rqState()` on them. Everything else on the page is untouched.

```js
// Plain jQuery — still works exactly as before
$('.menu-toggle').on('click', function () {
  $(this).toggleClass('open');
});

// reQuery — reactive state on a specific widget
$('#cart-widget').rqState({ itemCount: 0, total: 0 });
```
